# Harness 零基础入门指南

> 下载一个 App，说一句话，你的软件就做好了。

---

## 三步上手

```
1. 打开 OpenCode    2. 输入 /harness     3. 说你想做什么        4. 看着它做出来
   ┌──────────┐      ┌──────────┐        ┌──────────────┐      ┌──────────────┐
   │  OpenCode │─►   │ /harness │  ───►  │ "做个记账App" │─►   │  你的软件 ❤️  │
   └──────────┘      └──────────┘        └──────────────┘      └──────────────┘
```

就这三步。下面展开说说每一步怎么做。

---

## 1. 打开 OpenCode

### 1.1 安装

去 [opencode.ai](https://opencode.ai) 下载桌面客户端。macOS / Windows / Linux 都有。

双击安装，打开。跟装微信一样。

> 如果你更喜欢用终端，也可以命令行安装：
>
> ```bash
> curl -fsSL https://opencode.ai/install | bash
> ```

### 1.2 连接 AI

第一次打开 OpenCode 时，它需要一个「AI 大脑」来工作。

**推荐选择**：

| AI 提供商                | 特点                   | 适合               |
| ------------------------ | ---------------------- | ------------------ |
| **DeepSeek V4** ⭐ | 编程能力强、性价比极高 | **新手首选** |
| 智谱 GLM                 | 中文友好，便宜         | 中文项目           |
| Claude                   | 编程能力强             | 预算充足时         |
| GitHub Copilot           | GitHub 登录即用        | 已有 Copilot 账号  |

**怎么连**：在 OpenCode 里输入 `/connect`，选你要用的提供商，按提示注册登录。几分钟搞定。

> 推荐选 **DeepSeek V4**。编程能力第一梯队，价格只有 Claude 的几分之一。去 [platform.deepseek.com](https://platform.deepseek.com) 注册拿 API Key 就行。

### 1.3 启用 Harness

OpenCode 本身是个「裸」的 AI 程序员——聪明但没有纪律。Harness 让它变成一个**靠谱的** AI 程序员。

在 OpenCode 的输入框里输入斜杠命令：

```
/harness
```

这会加载 Harness 工程纪律系统。加载完成后，你的 AI 程序员就变成了一个有流程、有检查、有保障的工程团队。

> 💡 **每次新开 OpenCode 项目时**，输入一次 `/harness` 就行。之后的对话里 Harness 会自动工作，你不需要再管它。

### 1.4 打开你的项目文件夹

在 OpenCode 里打开你想放代码的文件夹（或者新建一个空文件夹）。

OpenCode 会自动分析项目结构，生成一份 `AGENTS.md` 帮它理解你的项目。

**到此为止，你的准备工作就结束了。** 没有别的要装，没有配置文件要改。

## 2. 告诉它你想做什么

### 2.1 你需要会的唯一技能

**用自然语言清楚描述你想要什么。**

就这么简单。你扮演产品经理，AI 扮演程序员。

**❌ 模糊的说法**：

> 做个网站

**✅ 清晰的说法**：

> 我想做一个个人记账的网页应用。用户可以记录每天的花费和收入，按类别看统计图表。数据保存在浏览器本地就行，不需要注册登录。界面要简洁好看。

### 2.2 两种模式：先想再做

OpenCode 有两个模式，按 **Tab 键**切换：

| 模式               | 图标 | 作用                                        |
| ------------------ | ---- | ------------------------------------------- |
| **计划模式** | 📋   | AI 只提方案，不改动代码。适合想清楚再动手。 |
| **构建模式** | 🔨   | AI 直接写代码、改文件。适合明确了就直接干。 |

**推荐流程**：先在计划模式里聊清楚，再切到构建模式让它实现。

### 2.3 怎么描述需求

一个好模板：

```
我想做 [什么]
给 [谁] 用的
需要能 [功能1]、[功能2]、[功能3]
不需要 [明确不做的]
参考 [你见过的类似产品]
```

举个例子：

> 我想做一个团队周报工具。
> 给 5-10 人的创业团队用。
> 每个人填写本周完成了什么、下周计划、需要帮助的事项。组长可以汇总查看。
> 不需要复杂的审批流程，越简单越好。
> 参考 Notion 的数据库视图。

### 2.4 跟 AI 聊天就是你的全部工作

发完需求之后，AI 可能会追问一些细节：

> 🤖 **AI**：这个周报工具，每个人填写的频率是每周一次吗？
> 🤖 **AI**：汇总视图需要导出功能吗（比如导出 PDF）？
> 🤖 **AI**：需要支持图片上传吗？

你只需要像聊天一样回答。不确定的就说「你帮我推荐」。

## 3. 看着它做出来

确认需求后，切到构建模式，说一句「开始做吧」。AI 就会：

```
📝 写需求文档 → 📐 规划方案 → ✂️ 拆分任务 → ⚡ 写代码 → ✅ 测试验收
```

全程自动。你喝杯咖啡就行。

### 3.1 中间会发生什么

AI 会在界面里展示它正在做的事情——创建文件、写代码、跑测试。你能实时看到。

偶尔它会停下来问你：

- 「这个功能涉及用户认证，属于高风险操作，你确认要这样设计吗？」
- 「我发现现有代码有一个冲突，你希望怎么处理？」

回答完它继续干。

### 3.2 做完了怎么看

AI 完成后会告诉你做了什么。你可以：

- 让 AI 打开应用看看效果
- 在浏览器里预览网页项目
- 让 AI 帮你检查做得对不对

### 3.3 不满意怎么办

直接说哪里不满意：

> 「统计图表的饼图颜色太少了，加更多颜色区分」

AI 会自动补上。

### 3.4 做坏了怎么办

OpenCode 有 `/undo` 命令——一键撤销上一步修改。

就像 Ctrl+Z 撤销打字一样，只不过撤销的是整个代码修改。

## 4. 出 Bug 了怎么办

软件做出来之后难免出问题。Harness 有一套专门的**调试流程**（七阶段）来系统性地解决。

### 4.1 直接告诉 AI 出了什么问题

不需要技术术语，用你的话说就行：

> 「点击删除按钮没反应」
> 「页面加载特别慢」
> 「添加了待办但是列表没显示」
> 「测试报错了，显示 xxx」

Harness 会自动切换到调试模式，创建一个**调试记录**（debug record），开始系统性地排查问题。

### 4.2 调试过程是怎样的

Harness 的调试不是「猜猜看」，而是**七阶段铁律流程**，每个阶段必须完成后才能进入下一个：

```
0. 登记  →  1. 复现  →  2. 收集证据  →  3. 诊断根因  →  4. 最小修复  →  5. 回归测试  →  6. 审查  →  7. 归档关闭
```

**三条铁律**：

| 铁律 | 含义 |
| ---- | ---- |
| 没有复现就不能修 | 必须先稳定复现问题，记录精确步骤 |
| 没有证据就不猜 | 先收集日志、报错、调用链，再提假设 |
| 三次修不好就升级 | 连续三次修复失败 → 停下来找你讨论，可能架构有问题 |

**每个阶段做什么**：

| 阶段 | AI 做什么 | 输出 |
| ---- | --------- | ---- |
| **登记** | 创建调试记录，记录症状、期望行为、实际行为 | `maintenance/debug/YYYY-MM-DD-描述/record.md` |
| **复现** | 用精确步骤稳定触发问题。复现不了就继续收集证据，不猜测 | 复现步骤、是否稳定复现 |
| **收集证据** | 收集报错日志、代码差异、调用链。如果是多模块，在每个模块边界加诊断 | 证据文件（日志、堆栈、diff） |
| **诊断根因** | 找能正常工作的类似代码 → 对比差异 → 提一个明确假设 → 最小验证 | 根因记录 + 假设表 |
| **最小修复** | 先给风险分级 → 写一个失败测试 → 只修根因，不做顺便的重构 | 补丁 + 行为变更说明 |
| **回归测试** | 写自动测试：修之前失败，修之后通过 | 红-绿验证通过的测试 |
| **审查关闭** | 审查代码变更 → 确认没有多余改动 → 按风险等级归档 | 审查清单 + 关闭记录 |

### 4.3 你需要做什么

大部分时候你什么都不用做。AI 自动走完全部七个阶段。

但遇到这些情况，AI 会**停下来问你**：

**情况一：高风险模块**

> 🤖 **AI**：这个 bug 涉及用户认证模块，属于高风险（core 级别）。我定位到的原因是 xxx，准备用 yyy 方式修复。你确认可以继续吗？

**情况二：连续修复失败**

> 🤖 **AI**：我已经尝试了 3 次修复都失败了。这说明可能不是代码层面的 bug，而是架构设计有问题。我建议我们讨论一下整体方案。

**情况三：无法复现**

> 🤖 **AI**：我按照描述的步骤试了很多次，但无法稳定复现这个问题。我加了监控日志，下次出现时可以收集更多信息。

你只需要说「继续」「换个方案」或「先别动」。

### 4.4 Bug 修复有保障

| 保障 | 说明 |
| ---- | ---- |
| **回归测试** | 每个 bug 修复都必须有自动测试，确保同一个 bug 不会再出现 |
| **红-绿验证** | 测试先失败（红）→ 修复后通过（绿），证明测试真的有效 |
| **风险评估** | 修复前后各做一次风险分级，确保修复没有引入新问题 |
| **最小修复** | 只修根因，不做顺便的重构、格式化、依赖升级 |
| **失败升级** | 三次修不好自动升级，不会死磕硬改 |
| **评估报告** | 中高风险（branch/core/infra）的修复会额外生成评估报告 |

## 5. 想做更大的项目？用 PM 流程

如果你不只是做一个小工具，而是想做一个**真正的产品**（比如一个有多个功能的 SaaS 应用），Harness 有一个**PM（产品经理）循环**帮你管好整个过程。

### 5.1 什么时候用 PM 流程

| 场景 | 用什么 |
| ---- | ------ |
| 做一个小功能、修个 bug | 直接告诉 AI，自动处理 |
| 做一个有 5+ 个功能的产品 | 输入 `/goal`，启动 PM 循环 |

**简单说**：小活直接说，大活用 PM。

### 5.2 PM 循环是什么

PM 循环 = 一个 AI 产品经理 + 一个 AI 程序员，**循环迭代**把你的想法变成成品。

```
你（老板） → AI 产品经理（理解需求、拆任务） → AI 程序员（写代码、跑测试） → 你验收 → 下一轮
```

你只需要：

1. **说你的想法**：像跟产品经理聊天一样
2. **回答问题**：AI 产品经理会追问细节
3. **做决策**：遇到选择时拍板
4. **看结果**：每轮完成后验收

### 5.3 PM 循环怎么工作

输入 `/goal` 后，AI 产品经理会进入一个**8 步循环**，每轮做一件事：

```
观察现状 → 检查准备 → 决定下一步 → 写任务单 → 分配给程序员 → 审查结果 → 更新状态 → 继续/停止
```

**每一步做什么**：

| 步骤 | AI 产品经理做什么 | 你需要做什么 |
| ---- | ----------------- | ------------ |
| **观察现状** | 读取项目状态、进度、上一轮结果 | 无 |
| **检查准备** | 确认需求文档、技术路线是否就绪 | 无（如果缺信息会问你） |
| **决定下一步** | 选一件事做：补充需求、做技术验证、分配开发任务、审查结果 | 无 |
| **写任务单** | 写一份清晰的任务描述：目标、范围、验收标准、禁止做的事 | 无 |
| **分配给程序员** | 把任务交给 AI 程序员执行 | 无 |
| **审查结果** | 检查代码变更、测试结果、有没有越界 | 无 |
| **更新状态** | 更新项目进度、循环日志 | 无 |
| **继续/停止** | 判断是否需要下一轮，还是停下来问你 | 偶尔回答问题 |

### 5.4 PM 循环的安全机制

| 机制 | 说明 |
| ---- | ---- |
| **迭代上限** | 默认有最大循环次数，不会无限跑 |
| **连续失败熔断** | 连续失败 3 次自动停下来找你 |
| **准备检查** | 需求没写清楚不会开始写代码 |
| **风险分级** | 高风险操作（改数据库、认证等）必须你批准 |
| **范围约束** | 每个任务单都有「禁止做的事」，防止 AI 擅自扩大范围 |

### 5.5 PM 循环中你会遇到的交互

**产品定义阶段**——AI 会追问：

> 🤖 **AI PM**：你这个记账应用，需要多人协作吗？
> 🤖 **AI PM**：数据是保存在浏览器还是需要云同步？
> 🤖 **AI PM**：有没有你喜欢的参考产品？

**技术验证阶段**——AI 会探索技术方案：

> 🤖 **AI PM**：我在验证一个技术方案，看看能不能用 localStorage 实现离线存储……

**开发阶段**——全自动，你看进度就行

**验收阶段**——AI 展示结果：

> 🤖 **AI PM**：这轮完成了「添加记账」功能。测试全部通过，代码变更 3 个文件。你可以看看效果。

**遇到选择**——AI 找你拍板：

> 🤖 **AI PM**：有两种方案实现图表功能。方案 A 用免费库但交互简单；方案 B 用付费库但效果更好。你选哪个？

### 5.6 PM 循环的文件结构

PM 循环会在你的项目里创建一个 `.pm/` 文件夹来管理状态：

```
.pm/
├── stable/                  # 稳定文档（很少改）
│   ├── product.md           # 产品定义
│   ├── roadmap.md           # 路线图
│   └── ux.md                # 用户体验方向（如果有界面）
└── runtime/                 # 运行时状态（每轮更新）
    ├── state.yaml           # 当前状态
    ├── next-task.md         # 当前任务单
    ├── worker-report.md     # 程序员的工作报告
    ├── acceptance-review.md # PM 的验收审查
    └── loop-control         # 循环控制（继续/停止/需要你决策）
```

**你不需要看这些文件**——它们是 AI 之间沟通用的。但如果你想深入了解进度，随时可以打开看。

### 5.7 什么时候 PM 循环会停下来

| 情况 | 说明 |
| ---- | ---- |
| **阶段完成** | 当前阶段的目标达成了，进入下一阶段 |
| **需要你决策** | 遇到产品/技术选择，AI 不能替你决定 |
| **遇到阻塞** | 外部依赖卡住了（比如需要你提供 API Key） |
| **达到上限** | 循环次数到了，防止无限运行 |
| **连续失败** | 连续 3 次失败，AI 不会再硬试 |

---

上面的流程里，你没看到「Harness」这个词——但它一直在后台工作。

### 一句话解释

**Harness 是 AI 程序员的工作纪律系统。**

打个比方：

- AI = 一个超级聪明但容易走神的程序员
- Harness = 一个严格的项目经理，盯住 AI 每一步都做对

没有 Harness，AI 可能会跳过测试、忘掉边界情况、写完才发现方向错了。

有 Harness，AI 必须：先想清楚 → 按计划做 → 做完检查 → 出报告。

### Harness 自动保护你

你不需要手动触发 Harness。它会根据你的修改自动判断风险：

| 风险级别 | 什么时候                 | 发生什么                      |
| -------- | ------------------------ | ----------------------------- |
| 🟢 低    | 改文档、修错别字、加测试 | AI 自己搞定                   |
| 🟡 中    | 做新功能、修功能 Bug     | AI 走完整流程，自动推进       |
| 🔴 高    | 改用户认证、核心逻辑     | **AI 停下来问你**       |
| ⛔ 极高  | 改数据库、部署配置、密钥 | **AI 必须你批准才能动** |

> 小事 AI 自己处理。大事 AI 来找你确认。你不需要盯着每一步。

### 怎么让 Harness 为你工作

你只需要**像平时一样说话**。Harness 会自动识别你在做什么：

| 你说的话                | Harness 自动做什么                       |
| ----------------------- | ---------------------------------------- |
| 「我有个想法，想做 XX」 | 追问细节 → 写需求 → 做计划 → 实现 |
| 「帮我做这个功能」      | 走规格→计划→任务→实现全流程           |
| 「这里有个 bug」        | 七阶段调试：登记→复现→证据→诊断→修复→回归→审查→归档 |
| 「页面卡/慢」           | 调试流程：收集证据→诊断→修复      |
| 「做得怎么样了」        | 生成评估报告                             |
| 「项目结构有点乱」      | 分析架构，给优化建议                     |
| 「这段代码重构一下」    | 风险评估→计划→实施→验证           |
| 「/goal 我想做个产品」  | PM 循环：需求→路线图→迭代开发→验收   |

## 实战：做一个待办事项应用

走一遍完整流程。

### 打开 OpenCode，开始对话

在 OpenCode 的输入框里打字：

> /harness 我想做一个待办事项网页应用。
> 用户可以添加待办、标记完成、删除。
> 数据保存在浏览器本地，不需要登录。
> 界面要好看，手机和电脑都能用。

### AI 追问，你回答

```
🤖 AI：需要给待办设置截止日期吗？
你：加上吧，截止日期快到的时候标红。

🤖 AI：需要分类功能吗？比如「工作」「生活」？
你：加上分类，用颜色区分。

🤖 AI：好的，我整理一下需求...
```

### 切到计划模式，审阅方案

按 **Tab** 切到计划模式。AI 会展示它的实现方案：

```
技术方案：
- 纯前端应用（HTML + CSS + JavaScript）
- 数据存储：浏览器 localStorage
- 界面：响应式布局，适配手机和电脑

文件结构：
- index.html — 主页面
- style.css — 样式
- app.js — 核心逻辑
```

看一眼，觉得 OK 就切到构建模式。

### 切到构建模式，开干

按 **Tab** 切到构建模式，说：

> 方案没问题，开始做吧

AI 就会自动创建文件、写代码、跑测试。你等着就行。

### 完成后查看

AI 做完后，告诉它：

> 打开浏览器让我看看效果

你的待办事项应用就出现在浏览器里了。

### 出了 Bug？

假设你发现「删除待办后，页面没更新」。直接告诉 AI：

> 「删除待办后页面没有刷新，待办还在列表里显示」

AI 会自动进入七阶段调试流程：

```
0. 登记：创建 maintenance/debug/2026-05-07-delete-not-refreshing/record.md
1. 复现：点击删除 → 确认数据删除了但列表没更新
2. 证据：数据层已删除但 UI 层未重渲染，缺少重新渲染调用
3. 诊断：对比添加功能的代码（正常刷新）→ 发现删除回调缺少刷新步骤
4. 修复：风险=leaf（低）→ 写失败测试 → 在删除回调中添加列表刷新
5. 回归：测试确认删除后列表更新（红→绿）
6. 审查：确认只改了删除回调，没有多余改动
7. 关闭：归档到 maintenance/debug/
```

全程自动，几秒搞定。

### 想加新功能？

继续在 OpenCode 里说：

> 加一个功能：待办可以设置优先级（高/中/低），高的排最前面

AI 自动走流程加上去。

## 常见问题

### 我完全不懂编程，能用吗？

**能。** 你的角色是「产品经理」——提需求、做决策、看结果。AI 负责所有技术细节。

### 我需要装什么？

只需要装 OpenCode。其他的（Python、Git、依赖库……）AI 会帮你装。

### AI 写的代码靠谱吗？

Harness 帮你把关：

- 每个功能都有**自动化测试**验证
- 每个修复都有**回归测试**防止旧病复发
- 有**需求文档**对照检查
- 高风险操作**必须你亲自批准**

### Bug 修完还会再出吗？

Harness 要求每个 bug 修复都必须有回归测试。同一个 bug 再出现的概率极低。

### 不满意怎么办？

直接告诉 AI 哪里不满意。或者用 `/undo` 撤销上一步。

### 出了问题怎么办？

告诉 AI：「出问题了，[描述你看到的错误]。」大部分问题 AI 可以自己修复。

如果修不好，在 OpenCode 里输入 `/undo` 回到上一个能用的状态。

### OpenCode 和 Cursor 有什么区别？

| 特点   | OpenCode                     | Cursor     |
| ------ | ---------------------------- | ---------- |
| 界面   | 桌面 App + 终端 + Web        | 代码编辑器 |
| 优势   | 开源、可装插件（如 Harness） | 所见即所得 |
| 装插件 | ✅ 支持                      | ✅ 支持    |

两个可以一起用。OpenCode 做架构和自动化，Cursor 做细节微调。

## 术语速查

遇到不认识的词，在这里查。

| 术语                      | 一句话解释                               |
| ------------------------- | ---------------------------------------- |
| **AI 编程助手**     | 你说话它写代码的工具                     |
| **OpenCode**        | 一个开源 AI 编程助手，有桌面客户端       |
| **Harness**         | AI 的工作纪律系统，确保每步做对          |
| **计划模式**        | AI 只提方案不改代码，用来看思路对不对    |
| **构建模式**        | AI 直接写代码改文件                      |
| **API Key**         | 连接 AI 服务的通行证                     |
| **风险级别**        | Harness 根据修改范围自动判断的严格程度   |
| **规格文档 (Spec)** | 你和 AI 确认的正式需求描述               |
| **测试 (Test)**     | 自动检查功能对不对的程序                 |
| **回归测试**        | 修完 bug 后的验证——确保同一个问题不再出现 |
| **TDD**             | 先定怎么验证，再动手做                   |
| **DDD**             | 按业务领域组织代码，不让代码变成一团乱麻 |
| **根因**            | 导致 bug 的真正原因（不是表面症状）      |
| **调试流程**        | Harness 的修 bug 流程：登记→复现→证据→诊断→修复→回归→审查→归档 |
| **PM 循环**         | AI 产品经理的迭代循环：观察→准备→决策→任务→执行→审查→更新→继续 |
| **任务单**          | PM 写给程序员的开发任务描述（目标、范围、验收标准） |
| **铁律**            | 调试流程中不可违反的硬性规则（如：没有复现就不能修） |
| **熔断**            | 连续失败时自动停止，防止死磕硬改         |
| **红-绿验证**       | 测试先失败（红）→ 修复后通过（绿），证明测试有效 |
